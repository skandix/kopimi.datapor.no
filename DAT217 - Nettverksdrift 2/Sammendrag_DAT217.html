Subnetting

  • IP nummer (v4, RFC 791)
  • Formen er 32 bit, skrevet slik 255.0.255.0
  • Binært: 11111111.00000000.11111111.00000000
    • Gir mulige adresser 256^4 = 4 294 967 296
  • Både nettverksadresse og maskin har adresse i samme format
    • A nett
      • Adresser som starter med 0 (0-127).0.0.0
      • første 8bit nettverks adresse 128 Stk
      • resterende 24 til hostid 16777216 Stk
    • B nett
      • Adresser som starter med 10(128-191).0.0.0
      • 16 bit til nettverksadresse 65536 Stk
      • 16 til hostid 65536 Stk
    • C nett
      • Adresser som starter med 110 (192-223).0.0.0
      • 24 bit til nettverksadresse 16777216 Stk
      • 8 bit til hostid 128 Stk

Den første adressen er nettverksadressen
Den siste er broadcast - kan nå alle på en gang

06 - DNS

DNS TCP/UDP port 53

Navnegiving av maskiner

  • Maskiner bruker binære tall

      Maskinen         |1000|0000|0010|0111|1100|1000|0000|0001|
  Grupperte 4 bit  |1000|0000|0010|0111|1100|1000|0000|0001|
  Heksadesimalt    |  8 |  0 |  2 |  7 |  C |  8 |  0 |  1 |
  Desimalt byte    |   128.  |    39.  |   200.  |    1    |

  • Navn grm-core-gw.uia.no

  • Mennesker liker best å huske navn

DNS - Domain Name System

  • Behov or forenkling av sammenheng mellom nummer og navn
    • Liste på hver maskin
      • Windows - C:\windows\system32\drivers\etc\hosts
      • Linux - /etc/hosts
        • I den spede begynnelsen (og i små nettverk)
    • Liste på én maskin som alle kan slå opp i
      • 1974 - ARPANET (hosts.txt)
    • Distribuert liste med distribuert ansvar for oppdatering.
      • Slik vi kjenner DNS i dag.

Domenenavn

  • Skiller nivå med punktum ‘.’
  • Toppen av rota er namespace ’ ’
  • Begynner med innerst og rota til slutt
    • Det er platt til 127 nivå på navnet
    • Hvert navn kan ikke være lengre enn 63 tegn
    • Totalt navn må ikke overskrive 254 tegn
    • Fullstendig navn har med .’ ’ til slutt
    • Kan utelate slutt, og program vil fulføre det.

Navnetjenere

  • Tar imot forespørsler fra resolvere, dvs klienter
  • Inneholder bl.a
    • IP nummer (A)
      • IPv6 nummer(AAAA)
    • Navn
      • Navnetjener (NS)
      • E-Post tjener (MX)
      • Alias (CNAME)
      • Autoritet (SOA)
      • Pekere (PTR)
      • Tidsgrenser for varighet av data (TTL)

Resolver - klienten

  • program på klienter
    • resolv.conf i /etc - Lages normalt automatisk (DHCP)
      • inneholder (IP)adresse til de navntjenerene som brukes
    • Retter forespørsler til en navntjener i bakgrunnen, slik at sammenheng mellom IP- nummer og navn blir ‘Gjenomsiktig’
    • Iterativ forespørsel
      • Gjentar forespørsel til svar foreligger
      • Begynner typisk fra toppen og ned
    • Rekursiv forespørsel
      • Ber tjeneren returnere svaret, dvs at det er tjeneren som foretar den iterative spørringen. Dette svaret gjemmes oftest i en cache på tjeneren.

Type navnetjenere

  • Innholdsnavnetjenere
    • Autorative - navnetjenere(Primary master - ns)
    • Slaver - fungerer som backup (Secondary M - ns)
      • Svarer som regel på iterative forspørsler fra underståendee
  • Cachende navnetjenere
    • Svarer på rekursive forespørsler
    • Sender svar tilbake til klient og cacher opplysningen i tilfelle ny forespørsel kommer

Kommandoer brukt i DNS

  • DIG
    • Anbefalt program
      • Gir gode svar og er pålitelig
      • Er lettest å bruke i automatiserte sammenhenger
  • NSLOOKUP
    • Ikke anbefalt
      • Finnes til windows
      • Har flere feie og mangler
  • HOST
    • Kan brukes
      • Hvis man ikke trenger tilleggsinformasjonen som man får ved dig
  • DRILL
    • fra NLnetLabs.nl
      • Noe flere funksjoner enn dig

Reverseoppslag

  • Hvis vi har nummer og vil finne navn kalles det reverseoppslag
  • Vi har toppdomener for adresser in-addr (ipv4)
  • Ved revers forespørsler snus nummeret og vi får navnet 4.3.2.1.in-adr.arpa. når vi spør etter adressen til ip-nr 1.2.3.4
  • Her finner vi en peker til navnet eg svar.datapor.no

08 - Mer om DNS

Oppgaver for soneansvarlige

  • Må ha kontroll over sin sone
  • Kontroll gis av foreldresonen til sonen
    • eg sonen grm.uia.no. må fåes av uia.no
  • En tjenermaskin kan betjene flere soner, og trenger ikke stå i sonen den betjener
  • Sone ansvarlige er som regel den tekniske ansvarlige
  • I norge er det norid.no som forvalter registreringer for .no domener (via registar)

Litt om ansvarlige

  • Det er en fordel å bruke mer enn ett tjenerprogram
  • Bør ha to (eller helst tre) server med sonefile, på forksjellige nett
  • Bruker man speisialtegn - ‘æ ø å’, bør man også registrere med eoa for robusthetens skyld
  • Bør også ha med reverseoppslag

DNSSEC

  • Er utvidelser av DNS for å sikre at data som gies er korrekte
  • Tanken er at det skal verifiseres at det virkelig er sonens ‘eier’ som gir data.
    • Hvis rotsonene er garantert ved public/private key, kan de godkjenne neste nivå, som igjen kan godkjenne nivået under det osv.
  • Prinsippet:
  • bruker private/public key
  • ‘Chain of trust’ med rot serverene på toppen
  • Tjenerene krypterer en hash av de sendte dataene med sin private nøkkel
  • Mottaker kan bekrefte ved å dekryptere hashen med den offentlige nøkkelen og sammenligne med en egen hash.
  • Har tatt tid å få rot og toppnivåene med.

Problemer med DNSSEC

  • Flere pakker blir større enn 512B => problem med udp.
    • Ikke alle brannmurer godtar TCP
    • Ikke alle brannmurer slipper store udp pakker inn
  • Problem å bytte nøkler synkront

NSD - Name Server Deamon

  • Er en autoritativ navnetjener
    • Rask, opp til 5 ganger raskere på DNSSEC svar
      • Sonefiler gjøres om til database med mulige svar istedet for å søke gjennom tekstfiler for å gi raskere svar.
    • Bygd etter 2003 for å være tilpasset DNSSEC
    • Alternativ til BIND også på rot-tjenerne

Unbound

  • Er en rekursiv navnetjener
    • Kan cache resultater
    • Kan utføre DNSSEC validering
    • Kan jobbe med IPv6
  • Laget av Nlnet labs (m.fl.) og er parhest til NSD
  • Påbegynt i 2004

Oppbygging av IPtables

  • Er et sett med tabeller
  • Hver tabell har ‘kjeder’
    • kjeden er en kjede av regler:
      • ‘Hvis pakkehodet ikke ser slik’ ut går man til neste regel
      • ‘Hvis pakkehoder, ser slik ut, gjør dette’
      • Hvis ingen av pakkene fyller betingelsene følges kjedens policy
    • Det er mulig å legge til egendefinerte kjeder
  • Kjeder som alt finnes
    • PREROUTING
    • INPUT
    • FORWARDING
    • OUTPUT
    • POSTROUTING

Tabeller

  • filter
    • er default tabell og brukes for filtrering av pakker
  • nat
    • brukes for å endre adresser (til og fra)
  • mangle
    • brukes for å endre pakker på diverse vis
  • raw
    • brukes stort sett bare til å slå av tracking
  • security
    • brukes spesielt for MAC
      • Mandatory Access Control

Form på regler

  • Først angis en betingelse, deretter en handling (mål) hvis betingelsen oppfylles.

En enkel brannmur

  • Setter policy til å forkaste alle innkommende pakker
    • iptables -P INPUT -j DROP
  • Setter policy til å forkaste alle pakker som skal til andre maskiner
    • iptables -P FORWARD -j DROP
  • Lar bare etablerte forbindelser komme inn utenfra
    • iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
  • Slipper gjennom pakker fra localhost (samme maskin)
    • iptables -A INPUT -i lo -j ACCEPT
  • Slipper ut alle pakker fra egen maskin
    • iptables -A OUTPUT -o lo -j ACCEPT

NAT - Enkel ruting

  • Vi bruker tabellen nat med kjedene PREROUTING og POSTROUTING
  • SNAT (Source NAT) brukes for å endre adressen i det den går ut
    • iptables -t nat -A POSTROUTING -o eth1 -j SNAT --to x.x.x.x
  • MASQUERADE er bare et speisaltilfelle der maskinen tildeles IP dynamisk (sjekker egen IP først)
  • DNAT- (Destination NAT) - brukes for å endre mottakeradressee
    • iptables -t nat -A PREROUTING -p TCP --dport 80 -i eth0 -j DNAT --to y.y.y.y:8080

HTTP

Hyper
Text
Transfer
Protocol

  • HTTP er en applikasjons-nivå protokopp for distribuerte hypermedia informasjonssystemer.
  • HTTP brukes også som en generisk protokoll for kommunikasjon mellom ‘user agents’ og proxy/gatewayer til andre Internet systemer som bruker andre protkoller som SMTP,NNTP,FTP,Gopher og WAIS. På den måten git HTTP hypermedia tilgang til ressurser fra mange typer applikasjoner.
  • Er en forespørsel/respons protokoll
  • Er tilstandsløs

HTTP version 2 (HTTP/2)

  • Optimized expression of the semantics of the Hypertext Transfer Protocol
  • More efficient use of networking resources and a reduced perception of latency by
    • introducing header field compression
    • allowing multiple concurrent exchanges on the same connection
    • introduces unsolicted push of representations from server to clients
  • Is an alternative to, but does not obsolete, the HTTP/1.1 message syntax.
  • HTTP’s existing semantics remain unchanged.

16 - FTP - File Transfer protocol

Anonymous ftp

  • Normalt vil brukere som finnes på maskinen bli ledet til sitt hjemmeområde (eller en katalog på hjemmeområdet)
  • Bruker /etc/passwd og /et/group til autentisering
  • Vil ofte ha mulighet for andre brukere også
    • Derfor kan brukeren anonymous (og ftp) defineres
    • anonymous (og ftp) vil ofte kunne logge inn uten passord, og styres til egne kataloger

Passiv ftp

  • FTP bruker to porter.
    • En til kontroll
    • En til data
  • Dette skaper problemer med brannmurer, siden det er serveren (utenfra) som prøver å etablere dataporten
  • RFC 1579 gir løsningen; Passiv ftp
    • Tjeneren åpner en post hos seg og opplyser klienten om hvilken det er.
    • så kan klienten åpne forbindelsen også på denne porten

Sikkerhet

  • FTP i seg selv er ikke sikkert
    • Alt overføres i klartekst
  • Sikkerhetstillegg finnes
  • I praksis:
    • Bedre autentiseringsmekanismer
    • Sikre FTP vha kryptering “pakket rundt” protokollen (SSL/TLS)

Metoder for å sikre ftp

  • FTPS (FTP Secure/FTP-SSL)
    • Sikrer FTP ved å kryptere data på et lavere OSI-Modellag. SSL/TLS.
    • Public key sertifikat, som HTTP over SSL
  • Secure File Transfer Protocol (SFTP)
    • Egen protokoll, subsystem for SSH
  • Tunnellere FTP over SSH (SSH FTP)
    Dette er ikke det samme som SFTP
    • Beskytter ofte bare signaleringskanale, ikke dataoverføingskanalen

17 - Nagios

  • Kan brukes til å overvåke trafikk over de fleste protokolle og tilstand på hardware
  • Er bygd opp med moduler som fritt kan lages og utvide mulighetene
  • Kommersielt, men har fortsatt core som open-source

Overvåke Windows Maskiner

  • Må installere en klient på Windows Maskinen: NSClient++
  • check_nt
    • Kommando i commands.cfg
  • Må fjerne # fra linjen med windows.cfg i nagios.cfg
  • Definer host i windows.cfg (en pr maskin)

Definere hva som overvåkes

  • Legg til service i windows.cfg for hver parameter som overvåkes

  • Les dokumentasjon for flere eksempler

    define service{
    use generic-service
    host_name winserver ;navn på maskinen
    service_description Memory Usage
    check_command check_nt!MEMUSE!-w 80 -c 90
    ;-w står for warning
    ;-c står for kritisk,
    ;ved 80 og 90% bruk av minnet
}

Overvåke snmp enheter

  • skrivere og switcher har typis snmp installert
  • Må definere host og service omtrent som for windows, men nå i hhv printer.cfg og switch.cfg
  • Må slå opp i dokumentasjon for kommandoer

andre enheter

  • Offentlige tjenester som HTTP, POP3, IMAP, FTP, SSH osv.
  • Finnes i commands.cfg

Driftsfilosofi

  • NMS - Network Management System
    • Styre-/Kontrollkonsollet - tar imot og presenterer data for brukere

    • Protokoll som styrer hvordan tlgang til informasjon i MIB skal være
  • MIB - Management Information Base
    • Database med informasjon om nettverket
  • SMI - Structure of Management Information
    • Beskrivelse av data-struktur på en MIV
  • Agent
    • Hjelpere ute på nettverket som tilrettelegger enhetene slik at informasjonen blir lesbar og tilgjengelig for SNMP
  • ICMP - Internet Control Message Protocol
    • Teller pakker, Registreer feilsendinger og redirect - retter opp i feil i rutere.
  • RMON - Remote MONitoring
    • Samler inn informasjon uten forespørsel fra NMS
  • SMON - Switch MONitoring
    • utvidelse av RMON, Muliggjør overvåking av svitsjer

    • Classless Inter-Domain Routing

    • Har tatt virtualisering videre og tilbyr bare programet
  • VMM - Virtual Machine Monitor Hypervisor
    • Kan være koblet rett til hardwaren, som OS

  • QoS - Quality Of Service
    • tjeneste kvalitet, prioriterer pakker etter hvor mye båndbredde de trenger til å fungere optimalt.

Backup

    • Definere kvantitet og kvalitet på tjenestene
  • ITIL - Information Technology infrastructure Libary
